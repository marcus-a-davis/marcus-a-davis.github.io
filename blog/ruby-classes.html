Stay Classy, Ruby

Think about machines. Not a particular machine, but the idea of a machine. Unless it happened at some kind of abstract art event, you've probably never encountered a generic machine. Still, you would feel comfortable saying machines as a group can perform operations like starting, shutting down, or navigating. Further, machines can have states like temperature, location, etc. However the abstract concept of a machine can't actually have any of this functionality or states. For that you would need a particular instance of a machine.

Similarly, if you were to model this in a programming language, you could create a Machine class that would also be a mere abstraction. To have functionality you'd have to create a particular instance of the machine. Once you've done that you could have a machine with functionality, or methods in Ruby-speak. Any particular machine could then have an actual state like temperature which could be stored in a variable.

Modeling this in Ruby is pretty straightforward. Let's do it:

            1  class Machine
            2    def initialize(location, temperature)
            3      @location = location
            4      @temperature = temperature
            5    end
            6  end

Line 1 of this code defines the class Machine. Line 2 creates an "initialize" method, which is a fancy way for saying, any time a particular machine is created do this. That initialize method takes two mandatory parameters: location and temperature. This means any time you create a Machine you must pass in a location and temperature.

On lines 3 and 4 I've defined initialize such that when you pass in a location and temperature for a machine, there will be one and only one of those variables for each machine. That's what that "@" notation in front of each of the variables means, they are called instance variables. Instance variables can also be passed between different methods the class. Let's look at an example of that:

            1  class Machine
            2    def initialize(location, temperature)
            3      @location = location
            4      @temperature = temperature
            5    end
            6  
            7    def change_location
            8     @location = "White Bear"
            9    end
            10   
            11    def print_location
            12      puts @location
            13    end
            14 end

On line 7 I have defined a method change_location which will change the @location instance variable to the value "White Bear". One line 9 I've created a simple print_location method that will print the value of @location. Note that I can use the @location method from multiple methods because it is an instance variable. Let's create a new Machine and call those methods to see what happens:

            ...
            11    def print_location
            12      puts @location
            13    end
            14 end
            15 
            16 my_machine = Machine.new("Vancouver", 25)
            17 my_machine.print_location
            18 my_machine.change_location
            19 my_machine.print_location

Continuing with the above class, on line 16 I create a new Machine class with the "Class.new" syntax and insert the required parameters. On lines 17-19 I print the location, call my change_location method and print the location again. Let's see what that returns:

            => "Vancouver"
            => "White Bear"

As you can see, the initial print_location method simply prints whatever you initially entered as the location. However, after I called change_location method on my_machine, the @location variable has been changed and as a result it prints "White Bear". Were I to create a new instance of the Machine class, the @location variable would be set to whatever I entered as its location. Like this:

            ...
            11    def print_location
            12      puts @location
            13    end
            14 end
            15 
            16 my_machine = Machine.new("Vancouver", 25)
            17 my_machine.print_location
            18 my_machine.change_location
            19 my_machine.print_location
            20
            21 new_machine = Machine.new("Chicago", 30)
            22 new_machine.print_location
            23 my_machine.print_location
            24 new_machine.print_location

What do you think that will print? Let's see:

            => "Vancouver"
            => "White Bear"
            => "Chicago"
            => "White Bear"
            => "Chicago"

As you can see, new_machine was unaffected by previous changes made to my_machine. As a result when I first call print_location on it, it returns the location I entered when I created it. If you think about these as real objects this makes sense. If you changed the license plate on a car that you already have, this wouldn't change the license plate on a different car.

Hopefully you can see that Ruby makes creating new classes of objects easy. You can also keep track of whether a particular instance of that object will be affected by a change by thinking about them as real objects. That's why modeling real objects like machines is a good way to get your head around class structures without getting bogged down in the details of any particular program.
